# Define and build shared building blocks for Topps programmes and collecting quests

**Goal:** Enable building blocks for Topps programmes

## What
Identify, define, and prioritize the fundamental building blocks that support both programmes (like FanCash Trading Card) and collecting quests (like Chase the Grail). Ensure that what we build for FanCash is designed with reusability in mind, so that quests can be launched on the same infrastructure.

## Building blocks to define

- [ ] **Discovery & education** — public-facing page that evolves pre/during/post
- [ ] **Redemption / entry** — configurable entry mechanism through Collect
- [ ] **Progress tracking** — dynamic view of participation status and what's remaining
- [ ] **Notifications & lifecycle** — triggered communications at key moments
- [ ] **Social & community** — leaderboards, other participants, shareable moments
- [ ] **Rewards & outcomes engine** — configurable reward system for different payoff types
- [ ] **Collection integration** — surfacing programme/quest participation within collections
- [ ] **Rules & eligibility engine** — configurable rules for participation, qualifying actions, time windows

## Key questions
- Which building blocks already exist (or are being built) for FanCash that we can reuse?
- Which need to be built new for quests?
- What level of configurability does each block need to serve both programmes and quests?
- How do we sequence this — what do we build first with FanCash, knowing quests follow?
- Who are the stakeholders for each building block (design, eng, product, ops)?

## Next steps
1. Map each building block against what's already in the FanCash V1 scope
2. Identify gaps — what FanCash doesn't cover that quests will need
3. Prioritize: which blocks give us the most reuse value if built right the first time
4. Align with eng on configurability requirements
5. Create a stakeholder positioning doc for the "building blocks" approach
